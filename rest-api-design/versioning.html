Versioning Strategies

How to evolve an API without breaking existing clients — the most consequential architectural decision you'll make.

StrategyMechanismExamplesProsConsVerdict
URL path /v1/users Google, PayPal, most public APIs Simple, browser-explorable, cache-friendly URL pollution; code duplication across versions Common
Header versioning API-Version: 2 Microsoft Azure Clean URLs Hard to test in browser; not cache-friendly without Vary Context-dependent
Query param ?api-version=2 Some Microsoft APIs Easy to add Pollutes query string; easy to overlook Avoid
Date-based Version = release date; pinned per account Stripe (2017-05-25), Shopify (2024-04) Incremental upgrades; no major jumps; 100+ changes supported by Stripe Complex version-change module architecture required Gold standard
Media type Accept: application/vnd.api+json;version=2 Zalando (preferred), GitHub (partial) RESTfully purist; clean URLs Complex for clients; not browser-explorable Niche
No versioning Only backwards-compatible changes forever Zalando (ideal), mature internal APIs Zero client disruption; lowest maintenance Requires extreme discipline; not always possible Ideal if achievable
Stripe's insight: Their version-change module architecture — each breaking change is a self-contained transform — lets them maintain compatibility with every version since 2011 (~100 changes) at fixed maintenance cost. The key: immutable response shapes per version, transformation in middleware.
