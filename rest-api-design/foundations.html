Foundational Theory & RFCs

REST is an architectural style derived incrementally by applying constraints to a null-style architecture — from Roy Fielding's 2000 doctoral dissertation. Most APIs labeled "REST" violate at least one constraint. Understanding the theory clarifies what trade-offs you're actually making.

Roy Fielding's Dissertation — Chapter 5: REST (2000)
Roy Thomas Fielding · UC Irvine

REST was designed for the web's architecture, not as a recipe for JSON APIs. Fielding derived it by starting from a "null style" and adding constraints, each providing specific properties. The 6 constraints work as a system — removing any one changes the resulting properties fundamentally.

REST's 6 Architectural Constraints

ConstraintWhat it meansBenefitTrade-off
Client-ServerSeparate UI concerns from data storagePortability + scalability through separation of concernsAdds coordination complexity
StatelessEach request contains all info needed; no server-side sessionVisibility, reliability, scalability — monitoring needs only one requestRepetitive data in requests; less server control over consistency
CacheableResponses declare whether they can be reusedReduces latency and network load significantlyRisk of stale data diverging from server state
Uniform InterfaceStandardized interactions: resource IDs, representations, self-descriptive messages, HATEOASSimplifies architecture; improves interaction visibilityDegrades efficiency — standardized formats aren't optimized per application
Layered SystemComponents only interact with adjacent layers; clients can't tell if directly connected to serverEnables load balancers, caches, security gateways, legacy integrationAdds latency and processing overhead
Code on Demand (optional)Servers can transfer executable code to extend client functionalityReduces features pre-implemented in clientsDecreases visibility; optional due to trust boundaries

Fielding's 6 Rules for REST APIs (2008)

  • APIs should not depend on a single communication protocol
  • Must not introduce non-standard protocol elements
  • Primary design effort should go into media types and relation names, not URI patterns
  • No fixed URI hierarchies — servers must control their own namespace
  • Resource types are invisible to clients; only media types and relations matter
  • Clients must enter with only an initial URI and standard media types; all state transitions driven by server-provided choices (HATEOAS)

Bottom line from Fielding: "If the engine of application state is not being driven by hypertext, then it cannot be RESTful. Period." Most industry APIs are RPC over HTTP, not REST.

Key RFCs

  • RFC 9110 — HTTP Semantics (2022, supersedes RFC 7231): method definitions, status codes, headers, content negotiation. The authoritative reference for HTTP behavior. Safe methods (GET, HEAD) enable caching; idempotent methods (PUT, DELETE) enable retry safety.
  • RFC 9457 — Problem Details for HTTP APIs (supersedes RFC 7807): standard application/problem+json error format. Fields: type (URI), title, status, detail, instance. Extensible with custom members.
  • RFC 8288 — Web Linking (supersedes RFC 5988): Link header for typed hypermedia relationships. Foundation that HATEOAS implementations (HAL, Siren) build upon. Used by GitHub for pagination (rel="next", rel="last").
  • RFC 6585 — Additional HTTP Status Codes: defines 429 Too Many Requests with Retry-After header — the standard for rate limiting responses.
  • RFC 6902 — JSON Patch: fine-grained PATCH operations (add, remove, replace, move, copy, test).
  • RFC 7386 — JSON Merge Patch: simpler PATCH format where null means delete a field.
  • RFC 6750 — Bearer Token usage in OAuth 2.0.
