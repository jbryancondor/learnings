Authentication Patterns

How clients prove identity — and how to choose the right mechanism.

API Keys
  • Passed as header (X-API-Key) or query param
  • Simple server-to-server auth
  • Static — no expiration or scope
  • Best practice: header only, never in URLs or client-side code
OAuth 2.0 + Bearer Tokens
  • Industry standard for delegated authorization
  • Authorization Code (server-side apps) vs Client Credentials (machine-to-machine)
  • Scoped access, expiration, refresh tokens
  • Used by: GitHub, Google, Microsoft, Slack, Atlassian
JWT (JSON Web Tokens)
  • Self-contained tokens with claims (sub, exp, iss, scopes)
  • Stateless verification — no DB lookup on each request
  • Best practice: short lifetime (15-60 min) + refresh tokens
  • Cannot be revoked individually until expiry
Advanced Patterns
  • mTLS: Mutual TLS — cryptographic client identity, zero-trust architectures
  • DPoP: Demonstration of Proof of Possession — binds tokens to clients, prevents replay
  • Fine-grained scopes: GitHub fine-grained PATs, OAuth scopes per resource
Rule: Always HTTPS — throw hard errors, never redirect to HTTP (Vinay Sahni). Short-lived tokens + refresh tokens. API keys server-side only. Log access, monitor anomalies.
