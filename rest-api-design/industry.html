Industry Design Guides

Authoritative, opinionated REST API guidelines published by major tech organizations — each reflecting years of operating APIs at scale.

Google API Design Guide + AIPs
Google Cloud
  • Resource-oriented design (AIP-121): hierarchical resources with collections of same-type items. Three pillars: resources (nouns), standard methods (verbs), stateless interactions.
  • Standard Methods: Get, List, Create, Update (PATCH only), Delete — every resource should support these.
  • AIP-132 (List): must include next_page_token; page_size + page_token on all list requests.
  • AIP-157 (Partial Responses): FieldMask via ?fields= — "*" returns all, omitted defaults to all.
  • Anti-patterns: API structure mirroring DB schemas; cyclic resource references.
  • Resource names as links — use full URL paths, not bare integer IDs.

Microsoft REST API Guidelines
Microsoft
  • Versioning required from day one. Extensible enums (modelAsString: true) to add values without breaking clients.
  • Idempotency: all methods must be idempotent. POST made idempotent via Repeatability-Request-ID + Repeatability-First-Sent headers.
  • Structured errors: ErrorResponse → ErrorDetail with code, message, target, details[], innererror. Plus x-ms-error-code header.
  • Conditional requests: ETag/If-Match/If-None-Match for optimistic concurrency.
  • LROs (Long-Running Operations): return 202 Accepted + status monitor resource for polling.
  • Naming: kebab-case for URL segments, camelCase for query params, pluralized collection names.
  • Error philosophy: "Customer can diagnose their own problems without contacting support."

Zalando RESTful API Guidelines
Zalando Tech
  • API-First principle: define API spec before writing code; peer review early.
  • API as Product: treat APIs as business assets — focus on simplicity, usability, consistency.
  • Naming: snake_case for JSON properties; kebab-case for URL paths; UPPER_SNAKE_CASE for enums.
  • Versioning philosophy: avoid versioning via compatible extensions. If needed, prefer media-type versioning over URL path.
  • Errors: MUST use RFC 7807/9457 Problem JSON. Never expose stack traces.
  • Pagination: prefer cursor-based over offset-based.
  • MUST/SHOULD/MAY classification system — over 200 structured rules.
  • Robustness: "Be liberal in what you accept, be conservative in what you send."

Atlassian REST API Guidelines
Atlassian
  • Resource expansion: ?expand=space,attachments — parts of a resource not returned unless explicitly requested.
  • JSON as primary format; standard HTTP methods.
  • OAuth 2.0 recommended for production; API tokens over username/password.
  • Minimizes network traffic by defaulting to compact responses.

PayPal API Standards
PayPal
  • Foundation principles: loose coupling, encapsulation, stability, reusable services, contract-based.
  • HATEOAS support: hypermedia links in responses following REST methodology.
  • Standard HTTP headers and status codes with custom header conventions.
  • Full versioning and deprecation policy included.

Consensus across all 5 guides: Resource-oriented design with nouns (not verbs), standard HTTP methods mapped to CRUD, structured error responses, explicit versioning, and pagination as a first-class concern.
