Error Handling

Structured, machine-readable errors are as important as correct responses. RFC 9457 is the emerging standard.

RFC 9457 — Problem Details (the standard)

HTTP/1.1 422 Unprocessable Entity
Content-Type: application/problem+json

{
  "type": "https://api.example.com/errors/validation-failed",
  "title": "Validation Failed",
  "status": 422,
  "detail": "The 'email' field must be a valid email address.",
  "instance": "/logs/abc-1234",
  "invalid-params": [{ "name": "email", "reason": "not a valid email" }]
}
OrgFormatKey fieldsStandout feature
RFC 9457 application/problem+json type, title, status, detail, instance Extensible — custom members allowed
Stripe Top-level error object type, code, message, param, doc_url doc_url links directly to error docs — best DX
Google gRPC Status → HTTP code, message, details[] (typed) Typed detail types: BadRequest, RetryInfo, DebugInfo
Microsoft ErrorResponse → ErrorDetail code, message, target, details[], innererror Nested innererror; x-ms-error-code header
