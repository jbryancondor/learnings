Rate Limiting

Standard headers, algorithms, and client-side strategies for fair API usage.

Standard response headers

HeaderMeaningUsed by
X-RateLimit-LimitMax requests per windowGitHub, Twitter, Stripe
X-RateLimit-RemainingRequests remaining in current windowGitHub, Twitter, Stripe
X-RateLimit-ResetWhen the limit resets (epoch seconds or timestamp)GitHub, Twitter, Stripe
Retry-AfterSeconds to wait before retrying (RFC 6585)Standard

Algorithm comparison

AlgorithmMechanismBurst supportBest for
Token bucketTokens added at fixed rate, each request consumes oneYesAPIs that should allow occasional bursts
Sliding windowRolling time window, no boundary burstNoPrecision — most fair to users
Fixed windowCount resets at fixed intervalsBoundary burst possibleMemory-constrained systems
Cost-basedQueries charged by complexity, not countDepends on budgetGraphQL (Shopify)
Client strategy: Implement exponential backoff — wait 2^n seconds on failure. Add random jitter to prevent thundering herd. Read rate limit headers proactively, don't wait for 429. Centralize rate limit state in Redis for distributed systems (Stripe, GitHub pattern).
