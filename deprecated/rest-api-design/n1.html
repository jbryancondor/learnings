The N+1 Problem

Fetching a list of resources then making N additional requests for each item's related data. The most common REST performance anti-pattern.

Example: GET /orders returns 50 orders → 50 × GET /customers/{id} for each order's customer. Total: 51 requests instead of 1.
SolutionMechanismExampleUsed byDeduplicates?
Expandable objects Query param opt-in embedding ?expand[]=customer Stripe, Atlassian No
Compound documents Sideloaded included array ?include=comments.author JSON:API Yes
Sparse fieldsets Per-type field filtering ?fields[articles]=title,body JSON:API, Twitter N/A
Field masks FieldMask proto / query param ?fields=name,customer.email Google APIs N/A
Embed parameter Custom embed convention ?embed=customer.name Vinay Sahni, WordPress No
Best practice: Link by default, embed by exception. Use JSON:API ?include= when the same related object appears repeatedly in a list (deduplication matters). Use Stripe's expand[] for single-resource expansion. For extreme N+1 cases across many resource types, consider GraphQL.
