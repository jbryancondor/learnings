Maturity Levels

Richardson Maturity Model

Introduced by Leonard Richardson, codified by Martin Fowler — a 4-level framework to measure how REST-like an HTTP API is.

Key insight: Most production APIs operate at Level 2 (HTTP verbs + proper status codes). Level 3 (HATEOAS) is the theoretical ideal but rarely implemented in practice. Roy Fielding argues that without HATEOAS, an API is not truly REST.
Level 0 — The Swamp of POX (Plain Old XML)
Avoid

A single URI, a single HTTP method (usually POST), with different operations encoded in the request body. Classic SOAP/RPC style — HTTP is used purely as a transport.

POST /appointmentService
{ "action": "getSlots", "date": "2026-01-01" }

POST /appointmentService
{ "action": "bookSlot", "doctor": "dr-smith", "slot": "1400" }

Level 1 — Resources
Minimal

Multiple URIs identifying individual resources. Still uses POST for everything, but at least different things have different addresses.

POST /doctors/dr-smith           → get available slots
POST /slots/1234                 → book a specific slot
Martin Fowler — Richardson Maturity Model ↗
Level 2 — HTTP Verbs
Industry Standard

Resources + correct HTTP methods (GET, POST, PUT, PATCH, DELETE) + meaningful status codes. This is where the vast majority of production APIs sit — including Stripe, GitHub, and most public APIs.

GET  /doctors/dr-smith/slots?date=2026-01-01   → 200 OK, list slots
POST /slots                                    → 201 Created, book slot
DELETE /slots/1234                             → 204 No Content
  • GET is safe and idempotent — cacheable by proxies
  • POST creates; PUT/PATCH updates; DELETE removes
  • Correct status codes: 201 on create, 404 on not found, 409 on conflict

Level 3 — HATEOAS (Hypermedia Controls)
True REST

Responses include hypermedia links that tell the client what actions are possible next. The client only needs to know the entry point — it discovers everything else at runtime. Roy Fielding's position: "If the engine of application state is not being driven by hypertext, then it cannot be RESTful."

{
  "slot": { "id": "1234", "time": "14:00", "status": "available" },
  "_links": {
    "self":   { "href": "/slots/1234" },
    "book":   { "href": "/slots/1234/bookings", "method": "POST" },
    "doctor": { "href": "/doctors/dr-smith" }
  }
}
  • Clients are decoupled from URL structure — URLs can change without breaking clients
  • Enables API evolution without client rewrites
  • New capabilities advertised in responses — self-documenting
  • Most APIs stop at Level 2 for the reasons below

Why most production APIs don't implement HATEOAS:

  • Tooling gap: very few libraries exist to build or consume HATEOAS APIs properly
  • Client behavior: most consumers (mobile apps, SPAs) hardcode endpoints anyway — decoupling is theoretical
  • Payload bloat: HATEOAS URIs add significant response size
  • Compatibility: retrofitting onto existing APIs breaks hardcoded clients
  • Learning curve: team members need significant training to use link traversal correctly
  • The fundamental paradox: if clients hardcode behavior regardless, the decoupling benefit doesn't materialize

When HATEOAS is worth it: closed ecosystems where both client and server are controlled; APIs designed for long-term evolution with many diverse consumers who can genuinely adopt dynamic link traversal.
