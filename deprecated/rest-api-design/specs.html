Specifications

Formal Specifications

Standards that define how REST responses should be structured — especially for related resources, hypermedia, and compound documents.

JSON:API
JSON:API v1.1
  • Compound documents with a single included array (deduplicates related objects)
  • relationships section: type + id by default; embed via ?include=
  • Sparse fieldsets: ?fields[articles]=title,body
  • Best for APIs returning lists with many repeated related objects

HAL
HAL — Hyperlink Application Language
  • _links: hypermedia references (always present)
  • _embedded: opt-in inline data (server-controlled)
  • No deduplication — embedded objects may repeat
  • Best for hypermedia-first APIs committed to HATEOAS
HAL Specification ↗  ·  HAL vs HATEOAS ↗
OpenAPI Initiative
OpenAPI Specification (OAS) 3.1
  • Machine-readable API description format (YAML/JSON)
  • Defines endpoints, request/response schemas, authentication, examples
  • Enables: code generation, docs, mocking, testing, SDK generation
  • Backed by Zalando, Microsoft, Google, Stripe, and most major API providers
OAS 3.1 Spec ↗
Google Protobuf / gRPC
Protocol Buffers + AIP Standards
  • Google's internal API standard; many AIPs apply to REST too
  • FieldMask for partial responses (maps to ?fields= in REST)
  • Resource names as full URL paths (not bare IDs)
  • Standard methods: Get, List, Create, Update (PATCH), Delete

Kevin Swiber
Siren — Hypermedia with Actions
  • Addresses HAL's gap: adds explicit actions (HTTP method, URL, fields)
  • Supports task-based non-CRUD interfaces
  • Entities with class, properties, actions, sub-entities
  • Full H-Factor coverage
Siren spec (GitHub) ↗
Mike Amundsen
Collection+JSON
  • Designed for read/write collection management
  • Templates for client data submission (write operations)
  • Queries property for predefined search parameters
  • Inspired by Atom (RFC 4287) and AtomPub (RFC 5023)
