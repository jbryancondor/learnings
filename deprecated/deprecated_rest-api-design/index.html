Maturity Levels

Richardson Maturity Model

Introduced by Leonard Richardson, codified by Martin Fowler — a 4-level framework to measure how REST-like an HTTP API is.

Key insight: Most production APIs operate at Level 2 (HTTP verbs + proper status codes). Level 3 (HATEOAS) is the theoretical ideal but rarely implemented in practice. Roy Fielding argues that without HATEOAS, an API is not truly REST.
Level 0 — The Swamp of POX (Plain Old XML)
Avoid

A single URI, a single HTTP method (usually POST), with different operations encoded in the request body. Classic SOAP/RPC style — HTTP is used purely as a transport.

POST /appointmentService
{ "action": "getSlots", "date": "2026-01-01" }

POST /appointmentService
{ "action": "bookSlot", "doctor": "dr-smith", "slot": "1400" }
Martin Fowler — Richardson Maturity Model ↗
Level 1 — Resources
Minimal

Multiple URIs identifying individual resources. Still uses POST for everything, but at least different things have different addresses.

POST /doctors/dr-smith           → get available slots
POST /slots/1234                 → book a specific slot
Martin Fowler — Richardson Maturity Model ↗
Level 2 — HTTP Verbs
Industry Standard

Resources + correct HTTP methods (GET, POST, PUT, PATCH, DELETE) + meaningful status codes. This is where the vast majority of production APIs sit — including Stripe, GitHub, and most public APIs.

GET  /doctors/dr-smith/slots?date=2026-01-01   → 200 OK, list slots
POST /slots                                    → 201 Created, book slot
DELETE /slots/1234                             → 204 No Content
  • GET is safe and idempotent — cacheable by proxies
  • POST creates; PUT/PATCH updates; DELETE removes
  • Correct status codes: 201 on create, 404 on not found, 409 on conflict
Martin Fowler — Richardson Maturity Model ↗  ·  Wikipedia ↗
Level 3 — HATEOAS (Hypermedia Controls)
True REST

Responses include hypermedia links that tell the client what actions are possible next. The client only needs to know the entry point — it discovers everything else at runtime. Roy Fielding's position: "If the engine of application state is not being driven by hypertext, then it cannot be RESTful."

{
  "slot": { "id": "1234", "time": "14:00", "status": "available" },
  "_links": {
    "self":   { "href": "/slots/1234" },
    "book":   { "href": "/slots/1234/bookings", "method": "POST" },
    "doctor": { "href": "/doctors/dr-smith" }
  }
}
  • Clients are decoupled from URL structure — URLs can change without breaking clients
  • Enables API evolution without client rewrites
  • New capabilities advertised in responses — self-documenting
  • Most APIs stop at Level 2 for the reasons below

Why most production APIs don't implement HATEOAS:

  • Tooling gap: very few libraries exist to build or consume HATEOAS APIs properly
  • Client behavior: most consumers (mobile apps, SPAs) hardcode endpoints anyway — decoupling is theoretical
  • Payload bloat: HATEOAS URIs add significant response size
  • Compatibility: retrofitting onto existing APIs breaks hardcoded clients
  • Learning curve: team members need significant training to use link traversal correctly
  • The fundamental paradox: if clients hardcode behavior regardless, the decoupling benefit doesn't materialize

When HATEOAS is worth it: closed ecosystems where both client and server are controlled; APIs designed for long-term evolution with many diverse consumers who can genuinely adopt dynamic link traversal.

Theory

Foundational Theory & RFCs

REST is an architectural style derived incrementally by applying constraints to a null-style architecture — from Roy Fielding's 2000 doctoral dissertation. Most APIs labeled "REST" violate at least one constraint. Understanding the theory clarifies what trade-offs you're actually making.

Roy Fielding's Dissertation — Chapter 5: REST (2000)
Roy Thomas Fielding · UC Irvine

REST was designed for the web's architecture, not as a recipe for JSON APIs. Fielding derived it by starting from a "null style" and adding constraints, each providing specific properties. The 6 constraints work as a system — removing any one changes the resulting properties fundamentally.

Fielding's Dissertation — Chapter 5 ↗  ·  Two Bit History — "Misappropriated REST" ↗

REST's 6 Architectural Constraints

ConstraintWhat it meansBenefitTrade-off
Client-Server Separate UI concerns from data storage Portability + scalability through separation of concerns Adds coordination complexity
Stateless Each request contains all info needed; no server-side session Visibility, reliability, scalability — monitoring needs only one request Repetitive data in requests; less server control over consistency
Cacheable Responses declare whether they can be reused Reduces latency and network load significantly Risk of stale data diverging from server state
Uniform Interface Standardized interactions: resource IDs, representations, self-descriptive messages, HATEOAS Simplifies architecture; improves interaction visibility Degrades efficiency — standardized formats aren't optimized per application
Layered System Components only interact with adjacent layers; clients can't tell if directly connected to server Enables load balancers, caches, security gateways, legacy integration Adds latency and processing overhead
Code on Demand (optional) Servers can transfer executable code to extend client functionality Reduces features pre-implemented in clients Decreases visibility; optional due to trust boundaries

Fielding's 6 Rules for REST APIs (2008)

  • APIs should not depend on a single communication protocol
  • Must not introduce non-standard protocol elements
  • Primary design effort should go into media types and relation names, not URI patterns
  • No fixed URI hierarchies — servers must control their own namespace
  • Resource types are invisible to clients; only media types and relations matter
  • Clients must enter with only an initial URI and standard media types; all state transitions driven by server-provided choices (HATEOAS)

Bottom line from Fielding: "If the engine of application state is not being driven by hypertext, then it cannot be RESTful. Period." Most industry APIs are RPC over HTTP, not REST.

Key RFCs

  • RFC 9110 — HTTP Semantics (2022, supersedes RFC 7231): method definitions, status codes, headers, content negotiation. The authoritative reference for HTTP behavior. Safe methods (GET, HEAD) enable caching; idempotent methods (PUT, DELETE) enable retry safety.
  • RFC 9457 — Problem Details for HTTP APIs (supersedes RFC 7807): standard application/problem+json error format. Fields: type (URI), title, status, detail, instance. Extensible with custom members.
  • RFC 8288 — Web Linking (supersedes RFC 5988): Link header for typed hypermedia relationships. Foundation that HATEOAS implementations (HAL, Siren) build upon. Used by GitHub for pagination (rel="next", rel="last").
  • RFC 6585 — Additional HTTP Status Codes: defines 429 Too Many Requests with Retry-After header — the standard for rate limiting responses.
  • RFC 6902 — JSON Patch: fine-grained PATCH operations (add, remove, replace, move, copy, test).
  • RFC 7386 — JSON Merge Patch: simpler PATCH format where null means delete a field.
  • RFC 6750 — Bearer Token usage in OAuth 2.0.

Industry

Industry Design Guides

Authoritative, opinionated REST API guidelines published by major tech organizations — each reflecting years of operating APIs at scale.

Google API Design Guide + AIPs
Google Cloud
  • Resource-oriented design (AIP-121): hierarchical resources with collections of same-type items. Three pillars: resources (nouns), standard methods (verbs), stateless interactions.
  • Standard Methods: Get, List, Create, Update (PATCH only), Delete — every resource should support these.
  • AIP-132 (List): must include next_page_token; page_size + page_token on all list requests.
  • AIP-157 (Partial Responses): FieldMask via ?fields= — "*" returns all, omitted defaults to all.
  • Anti-patterns: API structure mirroring DB schemas; cyclic resource references.
  • Resource names as links — use full URL paths, not bare integer IDs.
Google API Design Guide ↗  ·  AIP-121 ↗  ·  AIP-157 ↗  ·  Links not Keys ↗
Microsoft REST API Guidelines
Microsoft
  • Versioning required from day one. Extensible enums (modelAsString: true) to add values without breaking clients.
  • Idempotency: all methods must be idempotent. POST made idempotent via Repeatability-Request-ID + Repeatability-First-Sent headers.
  • Structured errors: ErrorResponse → ErrorDetail with code, message, target, details[], innererror. Plus x-ms-error-code header.
  • Conditional requests: ETag/If-Match/If-None-Match for optimistic concurrency.
  • LROs (Long-Running Operations): return 202 Accepted + status monitor resource for polling.
  • Naming: kebab-case for URL segments, camelCase for query params, pluralized collection names.
  • Error philosophy: "Customer can diagnose their own problems without contacting support."
Microsoft API Guidelines (GitHub) ↗  ·  Azure Guidelines ↗
Zalando RESTful API Guidelines
Zalando Tech
  • API-First principle: define API spec before writing code; peer review early.
  • API as Product: treat APIs as business assets — focus on simplicity, usability, consistency.
  • Naming: snake_case for JSON properties; kebab-case for URL paths; UPPER_SNAKE_CASE for enums.
  • Versioning philosophy: avoid versioning via compatible extensions. If needed, prefer media-type versioning over URL path.
  • Errors: MUST use RFC 7807/9457 Problem JSON. Never expose stack traces.
  • Pagination: prefer cursor-based over offset-based.
  • MUST/SHOULD/MAY classification system — over 200 structured rules.
  • Robustness: "Be liberal in what you accept, be conservative in what you send."
Zalando Guidelines ↗  ·  GitHub ↗
Atlassian REST API Guidelines
Atlassian
  • Resource expansion: ?expand=space,attachments — parts of a resource not returned unless explicitly requested.
  • JSON as primary format; standard HTTP methods.
  • OAuth 2.0 recommended for production; API tokens over username/password.
  • Minimizes network traffic by defaulting to compact responses.
Atlassian Guidelines ↗
PayPal API Standards
PayPal
  • Foundation principles: loose coupling, encapsulation, stability, reusable services, contract-based.
  • HATEOAS support: hypermedia links in responses following REST methodology.
  • Standard HTTP headers and status codes with custom header conventions.
  • Full versioning and deprecation policy included.
PayPal API Style Guide ↗
Consensus across all 5 guides: Resource-oriented design with nouns (not verbs), standard HTTP methods mapped to CRUD, structured error responses, explicit versioning, and pagination as a first-class concern.
Real World

Real-World API Patterns

How leading companies have solved REST design challenges in production APIs used by millions of developers.

Stripe
The Gold Standard of REST
  • Expandable objects: ?expand[]=customer — related data in one request. Elegant N+1 solution.
  • Date-based versioning: pinned per account, override via Stripe-Version header. 100+ breaking changes, all still supported.
  • Idempotency keys: Idempotency-Key header caches result and returns on retry.
  • Internal governance: 20-page API design doc; cross-functional review for every endpoint.
Stripe API ↗  ·  Versioning ↗
GitHub
Pragmatic Embedding
  • Eager embedding: always-needed relations embedded inline (e.g. owner inside repository).
  • Link header pagination: RFC 8288 rel="next", rel="last".
  • Rate limit headers: X-RateLimit-Limit, -Remaining, -Reset. Returns 429 on exceed.
  • 5,000 req/hour for authenticated; 60 for unauthenticated.

Twilio
Consistency as a Feature
  • Consistent structure across all products — code feels familiar regardless of which API.
  • Documentation-driven design: write docs for sample apps first, then build.
  • Transversal API team for governance across product teams.
  • Treat docs like code: tested, reviewed before release.

Twitter / X
Fields & Expansions Pattern
  • ?fields= and ?expansions= for controlling response shape — similar to JSON:API sparse fieldsets.
  • Cursor-based pagination with next_token.
  • Tiered rate limits by access level (Free → Enterprise).

Versioning Strategies

How to evolve an API without breaking existing clients — the most consequential architectural decision you'll make.

Strategy Mechanism Examples Pros Cons Verdict
URL path /v1/users Google, PayPal, most public APIs Simple, browser-explorable, cache-friendly URL pollution; code duplication across versions Common
Header versioning API-Version: 2 Microsoft Azure Clean URLs Hard to test in browser; not cache-friendly without Vary Context-dependent
Query param ?api-version=2 Some Microsoft APIs Easy to add Pollutes query string; easy to overlook Avoid
Date-based Version = release date; pinned per account Stripe (2017-05-25), Shopify (2024-04) Incremental upgrades; no major jumps; 100+ changes supported by Stripe Complex version-change module architecture required Gold standard
Media type Accept: application/vnd.api+json;version=2 Zalando (preferred), GitHub (partial) RESTfully purist; clean URLs Complex for clients; not browser-explorable Niche
No versioning Only backwards-compatible changes forever Zalando (ideal), mature internal APIs Zero client disruption; lowest maintenance Requires extreme discipline; not always possible Ideal if achievable
Stripe's insight: Their version-change module architecture — each breaking change is a self-contained transform — lets them maintain compatibility with every version since 2011 (~100 changes) at fixed maintenance cost. The key: immutable response shapes per version, transformation in middleware.

Pagination Patterns

How to page through large result sets — a choice with significant performance and consistency implications.

Pattern Params DB Query Performance Consistency Random Access Best for
Offset ?offset=20&limit=10 LIMIT 10 OFFSET 20 Degrades on large offsets (full scan) Inconsistent if data changes between pages Yes Small, static datasets
Cursor (keyset) ?cursor=eyJpZCI6MTIzfQ== WHERE (created_at, id) > (?, ?) O(1) regardless of position Consistent under concurrent writes No Large, frequently-changing data
Page token (Google) ?page_token=TOKEN Server-controlled opaque token O(1) with index Server-controlled No Public APIs with server-side cursor management
Link header (RFC 8288) Encoded in Link response header Depends on underlying impl Depends Depends Depends Decoupled pagination URLs from response body (GitHub)
Industry consensus: Prefer cursor-based (keyset) pagination for production APIs. Zalando, Google, Slack, and Twitter all recommend it. Use indexed, immutable, unique fields for the cursor — typically a compound (created_at, id) pair. Always return a next_page_token / next link — never make clients construct pagination URLs.

Cursor implementation example

-- Efficient keyset pagination (no OFFSET)
SELECT * FROM orders
WHERE (created_at, id) < (:cursor_created_at, :cursor_id)
ORDER BY created_at DESC, id DESC
LIMIT 25;

-- Response includes
{
  "data": [...],
  "next_page_token": "eyJjcmVhdGVkX2F0IjoiMjAyNi0wMS0wMSIsImlkIjoiMTIzIn0="
}

Error Handling

Structured, machine-readable errors are as important as correct responses. RFC 9457 is the emerging standard.

RFC 9457 — Problem Details (the standard)

HTTP/1.1 422 Unprocessable Entity
Content-Type: application/problem+json

{
  "type": "https://api.example.com/errors/validation-failed",
  "title": "Validation Failed",
  "status": 422,
  "detail": "The 'email' field must be a valid email address.",

  "invalid-params": [{ "name": "email", "reason": "not a valid email" }]

Authentication Patterns

How clients prove identity — and how to choose the right mechanism.

API Keys
  • Passed as header (X-API-Key) or query param
  • Simple server-to-server auth
  • Static — no expiration or scope
  • Best practice: header only, never in URLs or client-side code
OAuth 2.0 + Bearer Tokens
  • Industry standard for delegated authorization
  • Authorization Code (server-side apps) vs Client Credentials (machine-to-machine)
  • Scoped access, expiration, refresh tokens
  • Used by: GitHub, Google, Microsoft, Slack, Atlassian
JWT (JSON Web Tokens)
  • Self-contained tokens with claims (sub, exp, iss, scopes)
  • Stateless verification — no DB lookup on each request
  • Best practice: short lifetime (15-60 min) + refresh tokens
  • Cannot be revoked individually until expiry
Advanced Patterns
  • mTLS: Mutual TLS — cryptographic client identity, zero-trust architectures
  • DPoP: Demonstration of Proof of Possession — binds tokens to clients, prevents replay
  • Fine-grained scopes: GitHub fine-grained PATs, OAuth scopes per resource
Rule: Always HTTPS — throw hard errors, never redirect to HTTP (Vinay Sahni). Short-lived tokens + refresh tokens. API keys server-side only. Log access, monitor anomalies.

The N+1 Problem

Fetching a list of resources then making N additional requests for each item's related data. The most common REST performance anti-pattern.

Example: GET /orders returns 50 orders → 50 × GET /customers/{id} for each order's customer. Total: 51 requests instead of 1.
Solution Mechanism Example Used by Deduplicates?
Expandable objects Query param opt-in embedding ?expand[]=customer Stripe, Atlassian No
Compound documents Sideloaded included array ?include=comments.author JSON:API Yes
Sparse fieldsets Per-type field filtering ?fields[articles]=title,body JSON:API, Twitter N/A
Field masks FieldMask proto / query param ?fields=name,customer.email Google APIs N/A
Embed parameter Custom embed convention ?embed=customer.name Vinay Sahni, WordPress No
Best practice: Link by default, embed by exception. Use JSON:API ?include= when the same related object appears repeatedly in a list (deduplication matters). Use Stripe's expand[] for single-resource expansion. For extreme N+1 cases across many resource types, consider GraphQL.

References

Books & Guides

The most authoritative books and practitioner guides on REST API design.

O'Reilly · 2013
RESTful Web APIs
Richardson, Amundsen & Ruby. Deep theoretical treatment of REST with strong emphasis on hypermedia. Richardson (who created the RMM) and Amundsen (who created Collection+JSON) cover the gap between theory and practice, advocating for real HATEOAS adoption.
theory hypermedia
O'Reilly · 2011
REST API Design Rulebook
Mark Massé. Prescriptive rule-based approach. Covers URI design (use hyphens, lowercase, no trailing slashes), HTTP method semantics, metadata headers, and the WRML framework. Follow rules as a complete set or à la carte.
practical rules   O'Reilly ↗
Phil Sturgeon · 2014
Build APIs You Won't Hate
End-to-end practical guide from planning to production. Covers endpoint theory, serialization, status codes, errors, authorization, versioning, and HATEOAS pragmatics. Opinionated and conversational — written by a developer who's made all the mistakes.
pragmatic popular   apisyouwonthate.com ↗
O'Reilly · 2018
Designing Web APIs
Jin, Sahni & Shevat. Covers request-response and event-driven paradigms, scaling, API-as-product thinking, and developer experience. Case studies from Slack, Stripe, Facebook, Microsoft, GitHub. Strong on the business side of API design.
modern dx   O'Reilly ↗
O'Reilly · 2010
RESTful Web Services Cookbook
Subbu Allamaraju (eBay platform architect). 100+ problem/solution recipes: URI design, HTTP methods, caching, security, transactions, batch processing, partial updates. Language-agnostic deep practical coverage.
cookbook patterns   O'Reilly ↗
Vinay Sahni · 2014
Best Practices for a Pragmatic RESTful API
The most-cited REST guide online. Opinionated, battle-tested decisions: nouns not verbs, snake_case (20% easier to read per research), SSL everywhere — throw hard errors never redirect, pretty-print by default, embed convention, rate limit headers.
vinaysahni.com ↗
